## [0.1.4] — 2026-05-26

Supply-chain hardening. The release pipeline now attaches the SBOM as an OCI referrer on the GHCR image rather than leaving it stranded as a CI workflow artefact. No solver, chart or runtime behaviour change — this is workflow-only — but the published image is the v0.1.4 build, and the release pipeline stamps the Helm chart that gets pushed to `oci://ghcr.io/xmv-solutions-gmbh/charts` with the tag version too (`helm package --version 0.1.4 --app-version 0.1.4`), so the canonical install pair is **chart 0.1.4 + image 0.1.4**. The in-tree `charts/cert-manager-webhook-hcloud-zones/Chart.yaml` still reads `version: 0.1.2 / appVersion: "0.1.3"` because the release workflow does the stamping at package time — do not be misled by the working-tree values.

### Fixed

- **`.github/workflows/release.yml`** (PR [#16](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/16)) — the SBOM job in v0.1.3 ran `anchore/sbom-action@v0` with `image:` and `registry-username:` / `registry-password:` set, which looks like it pushes the SBOM into the registry but does not: the action's only effect is uploading the SBOM as a GitHub Actions workflow artefact. Symptom on the v0.1.3 release: the image carried a cosign signature (`sha256-<digest>.sig` tag → HTTP 200) but no SBOM sidecar (GET `.sbom` tag → HTTP 404, OCI referrers API listed only the signature). The job is now split into two steps — `anchore/sbom-action` generates the SBOM to a file via `output-file:` (kept on purpose for minimal diff: same syft pin, same scanner config, same workflow-artefact upload for ad-hoc download), then `cosign attest --type spdxjson` attaches it as an OCI referrer of the image. After this fix: `cosign verify-attestation` returns the SBOM via Sigstore Rekor; `GET /v2/<repo>/referrers/sha256:<digest>` lists both the signature and the attestation; downstream supply-chain tools (Grype, Trivy, sigstore policy controllers) can resolve the SBOM from the image alone, no GitHub Actions artefact gymnastics required.

## [0.1.3] — 2026-05-26

Harness-blocker fix release. The v0.1.0 image, chart and harness all shipped working in isolation but failed against the real Let's Encrypt staging endpoint in three load-bearing ways plus two harness-side ways. Bundled as PR [#15](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/15) (five commits) because each fix on its own was a no-op without the others: a static binary that the framework's flag parser still rejects, or a parsed flag set on a binary the kernel cannot exec, or a webhook that runs but writes TXT records to the wrong RRSet, would each leave the LE-STAGING harness red. None of these were caught by the unit tests in v0.1.0 — the harness was the only thing that exercised the end-to-end real-cluster path, which is why **harness-first** is non-negotiable for this kind of integration.

Two chart-only intermediate pushes happened during the debugging session — chart `0.1.1` pinning to image `0.1.1`, chart `0.1.2` pinning to image `0.1.3`. Neither was tagged in git; both are superseded by the v0.1.3 image / chart `0.1.2` pair shipped here (and again by v0.1.4 above). Treat them as side-effects of the v0.1.3 release journey, not as independent releases.

### Fixed

- **`internal/solver` TXT record name now zone-relative** (PR [#15](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/15), commit `fix(solver): build TXT record name from FQDN relative to zone apex`) — `Present` / `CleanUp` / `reconcileExisting` hard-coded the RRSet name to `_acme-challenge`, so the webhook only worked for zone-apex certificates. For any subdomain cert (the common case — e.g. `app.example.com`) the TXT record was written to the apex name regardless of the actual challenge's `ResolvedFQDN`, and Let's Encrypt validation never found the expected response token. New `relativeRecordName(resolvedFQDN, zoneApex)` strips the zone suffix and returns the leaf-relative label cert-manager actually needs (`_acme-challenge.example.com.` → `_acme-challenge`; `_acme-challenge.app.example.com.` → `_acme-challenge.app`; `_acme-challenge.foo.bar.example.com.` → `_acme-challenge.foo.bar`). Defensive fallback to `_acme-challenge` if the router invariant is ever broken (FQDN does not end with the picked zone apex), so a future refactor surfaces the misconfiguration at the Hetzner API rather than silently writing to the apex. Surfaced by the LE-STAGING harness regression for `app-a-<run-id>.xmv-example.com` — the unit test in `solver_test.go` used `_acme-challenge.app.example.com.` as the FQDN but never asserted on the resulting record name, so the bug shipped through v0.1.0. The new `relative_name_test.go` pins eight name-mapping cases including the harness-style FQDN, and the multi-credential test in `solver_test.go` now asserts `rrset.Name == "_acme-challenge.app"`.
- **`Dockerfile` produces a fully static binary** (PR [#15](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/15), commit `fix(build): produce a fully static binary with CGO_ENABLED=0`) — v0.1.0's image shipped a glibc-dynamically-linked Go binary on top of `distroless/static-debian12`, which has no libc. The kernel could not locate the dynamic loader `/lib64/ld-linux-x86-64.so.2` and rejected the exec; container runtime surfaced this as the very misleading `exec /webhook: no such file or directory` even though `/webhook` was present in the image and the multi-arch manifest was correct. Setting `CGO_ENABLED=0` forces `go build` to emit a static binary that the static-distroless base can actually run. While here, two related buildx hygiene fixes: `--platform=$BUILDPLATFORM` on the build stage so cross-arch jobs run the Go toolchain natively on the host arch and only the linker output is cross-compiled (drops build time on multi-arch CI from minutes-per-arch under qemu to seconds); and `ARG TARGETOS` / `ARG TARGETARCH` passed to `go build` so the second-stage static binary actually targets the requested platform of the final image.
- **`cmd/cert-manager-webhook-hcloud-zones/main.go` drops `pflag.Parse()`** (PR [#15](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/15), commit `fix(cmd): let the framework own the flag set; drop pflag.Parse()`) — cert-manager's `RunWebhookServer` wraps a cobra command whose flag set defines the actual TLS / serve flags the chart passes via container args (`--tls-cert-file`, `--tls-private-key-file`, `--secure-port`, `--v`). Calling `pflag.Parse()` in our `main` BEFORE handing control to `RunWebhookServer` made the parser fail on the first unknown flag the chart passes — observed in the cluster as `unknown flag: --tls-cert-file` immediately followed by `exec /webhook: no such file or directory` (the second line was the CGO regression; both fired together). The canonical webhook-example reads `GROUP_NAME` from the environment exclusively; this binary now matches that — the env override was already supported via `envOrDefault`, the flag was redundant tooling that conflicted with the framework. The Helm chart already sets `GROUP_NAME` from `.Values.groupName`.
- **`tests/harness/test-apps/cluster-issuer.yaml` schema, valid LE email, lowercase `RUN_ID`** (PR [#15](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/15), commit `fix(harness): manifest schema, valid LE email, recursive DNS check, lowercase RUN_ID`) — four independent harness-side bugs the LE-STAGING run surfaced once the image-side bugs were fixed. (1) `cluster-issuer.yaml` carried solver entries with a top-level `apiTokenSecretRef` block; the webhook's `parseConfig` rejects that with `config has no credentials; at least one is required` because the supported schema is the `docs/app-concept.md` § 3.2 shape with a `credentials:` list. Rewritten to one solver block that carries both credentials (project-a and project-b), with zones, Secret refs and explicit namespaces — and no outer `selector.dnsZones`, which is redundant when the webhook routes by `zones` in its own config. (2) `groupName` was `acme.example.com` from the original template; must match the chart's `.Values.groupName` (default `acme.hcloud-zones.xmv.de`); fixed. (3) `email` was `harness@example.com` — Let's Encrypt rejects this with `400 invalidContact "forbidden domain example.com"`. Bumped to `harness@xmv.de`. (4) `RUN_ID` embedded the literal `T` from `date +%Y%m%dT%H%M%S`; RFC 1123 requires lowercase in DNS hostnames; the Ingress admission webhook rejected the rendered hosts with `must consist of lower case ...`. Switched to `%Y%m%dt%H%M%S`.

### Changed

- **`charts/cert-manager-webhook-hcloud-zones/Chart.yaml`** (PR [#15](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/15), commit `chore(release): bump chart to 0.1.2 / appVersion 0.1.3`) — chart pin moves to the rebuilt image that contains the three load-bearing fixes (CGO, pflag, subdomain TXT name). Chart `0.1.2` was packaged + pushed to `oci://ghcr.io/xmv-solutions-gmbh/charts` out-of-band of CI during the debugging session; the v0.1.4 release pipeline regenerates it under the canonical workflow as chart `0.1.4` + appVersion `0.1.4`.
- **`tests/harness/run.sh`** (PR [#15](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/15), part of the manifest-schema commit above) — `WEBHOOK_CHART_VERSION` bumped to `0.1.2`; the cert-manager install now passes `--dns01-recursive-nameservers-only=true`. The default authoritative-NS self-check hangs in an infinite `not yet propagated` loop against Hetzner Cloud DNS zones whose registry delegation points at Hetzner *Robot* DNS hostnames AND that carry a wildcard CNAME (the `xmv-solutions.de` zone in the harness fixture has exactly this shape). The recursive resolver path (Cloudflare + Quad9) sidesteps the pathology and is the configuration the README should recommend for any harness operator whose zones share that shape.
- **`internal/solver/solver.go` `const` block** (PR [#15](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/15), commit `chore(lint): gofmt-align the const block in solver.go`) — gofmt re-spaces the const block after the `defaultChallengeName` → `defaultChallengeLabel` rename pushed the new identifier past the shared column boundary. Pure whitespace; CI's golangci-lint regression-fix only.

## [0.1.0] — 2026-05-26

Initial release: webhook binary + Helm chart + GHCR release pipeline. This is Phase 2 of issue [#7](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/issues/7) — the first end-to-end deployable artefact set for the project. The release shipped with three latent bugs (CGO-dynamic-link, redundant `pflag.Parse()`, apex-only TXT record name) and two harness-side bugs (manifest schema, invalid LE email + `RUN_ID` casing) that were only caught once the LE-STAGING harness ran against a real cluster; all five are fixed in v0.1.3 above.

### Added

- **`.github/workflows/release.yml`** (PR [#11](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/11), part of [#7](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/issues/7)) — release pipeline triggered on every `v*` tag push. Builds a multi-arch container image (`linux/amd64` + `linux/arm64`) and pushes it to GHCR; signs the image with cosign keyless signatures via Sigstore; generates and attaches an SBOM (anchore/sbom-action) to the GHCR image (this is the step v0.1.4 had to fix — see above); packages the Helm chart and pushes it to `ghcr.io/<owner>/charts` as an OCI artefact; and creates a GitHub Release with auto-generated release notes. Image signing and SBOM attachment each have their own `id-token: write` grant for OIDC; the top-level `permissions` block is narrowed to `contents: read`. The `helm package` step stamps the chart with `--version "${VERSION#v}" --app-version "${VERSION#v}"`, so the chart artefact published to GHCR always matches the git tag even when the in-tree `Chart.yaml` lags behind.
- **`docs/app-concept.md`** (PR [#1](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/1)) — the foundational design spec: motivation (legacy Hetzner DNS Console vs. the new Hetzner Cloud Zones API), MVP scope, the load-bearing zone-to-token routing decision (multi-project from a single deployment), the configuration & deployment model, the full test strategy (unit / integration / harness with Let's Encrypt staging), operational + security posture, publication plan (GHCR image + OCI chart + GH Pages + ArtifactHub), and the ticket sequence to `v0.1.0`.
- **`tests/harness/test-apps/`** (PR [#4](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/4)) — the harness manifest set: `ClusterIssuer` (Let's Encrypt staging, two `apiTokenSecretRef` entries — one per Hetzner Cloud project, exercising the "one token, multiple zones" path for project B), three `Certificate` resources (one per harness zone), the accessory `Pod` / `Service` / `Ingress` backend, and a `kustomization.yaml`. Manifests carry `${RUN_ID}` / `${HARNESS_ZONE_*}` placeholders rendered by `envsubst` at harness runtime, so concurrent fires against the same zones cannot collide. The `ClusterIssuer` schema shipped here did not match what `parseConfig` expects — fixed in v0.1.3.
- **`tests/harness/run.sh`** (PR [#5](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/5)) — the bring-your-own-kubeconfig harness runner. Validates required env vars (`HARNESS_KUBECONFIG`, `HCLOUD_TOKEN_PROJECT_{A,B}`, `HARNESS_ZONE_{A,B1,B2}`), installs cert-manager and the webhook chart from its **published GHCR OCI artefact** (the production install path — not a local working-tree build), applies the test-app manifests, waits for all three `Certificate`s to reach `Ready=True` on a shared 10-minute budget, then asserts issuer is LE staging, SANs match the expected per-fire FQDN, and `tls.key` parses. Exit codes: `0` success / `1` setup failure / `2` assertion failure. `--cleanup` is opt-in and is honoured only when every assertion has passed — failed runs intentionally leave cluster state in place for inspection.
- **README "How to run the harness" section** — operator-facing quick reference for the harness: prerequisites, the six required env vars, invocation, exit codes, the failed-run-leaves-state principle, and the long-lived `hcloud-token-project-{a,b}` Secrets that persist between fires for cluster warmth.
- **`cmd/cert-manager-webhook-hcloud-zones/main.go`** (PR [#8](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/8), part of [#7](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/issues/7)) — the binary entry point: registers the `hcloud-zones` solver with the cert-manager webhook framework via `cmd.RunWebhookServer`. The group name defaults to `acme.hcloud-zones.cert-manager.io` and is overridable via `--group-name` or the `GROUP_NAME` environment variable. The `--group-name` flag and the `pflag.Parse()` call that backed it shipped here as redundant tooling that conflicted with the cert-manager webhook framework's own flag set — removed in v0.1.3.
- **`Dockerfile`** (PR [#8](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/8)) — multi-arch capable multi-stage image: Go build stage (`golang:1.25`) produces a trimmed, stripped binary; final stage is `gcr.io/distroless/static-debian12:nonroot` running as `nonroot:nonroot`. The CI builds for `linux/amd64` and `linux/arm64` by passing `--platform` at build time. Shipped without `CGO_ENABLED=0`, which made the resulting binary glibc-dynamic and therefore unrunnable on the static-distroless base — fixed in v0.1.3.
- **`charts/cert-manager-webhook-hcloud-zones/`** (PR [#10](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/10), part of [#7](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/issues/7)) — the Helm chart that the release pipeline packages and pushes to GHCR OCI. `Chart.yaml` shipped as `version: 0.1.0 / appVersion: "0.1.0"`; the release pipeline overrides both at `helm package` time to match the git tag, so the artefact on `oci://ghcr.io/xmv-solutions-gmbh/charts` for the v0.1.0 release is chart `0.1.0` + appVersion `0.1.0`.

### Changed

- **`README.md`** — replaced the generic template README with the project-specific one (one-sentence pitch, "Why this exists" framing the legacy-DNS-Console vs. Cloud-Zones gap, roadmap to `v0.1.0`, documentation index, the new harness section). Previously this slot held the `oss-project-template` README; PR [#1](https://github.com/XMV-Solutions-GmbH/cert-manager-webhook-hcloud-zones/pull/1) replaced it.

## [v0.3.0] — 2026-05-09

LLM-product-independence pass. The template no longer privileges any specific AI coding tool: every agent reads the same canonical brief, and the tool-specific files are minimal pointers. Plus a cleanup of GitHub-Copilot-Chat-specific shortcuts that were either redundant or anti-patterns.

### Added

- **`AGENTS.md`** as the canonical, tool-agnostic AI-agent brief at the repo root. Every coding agent reads the same file: Codex auto-discovers `AGENTS.md` natively; `CLAUDE.md` (Claude Code) and `.github/copilot-instructions.md` (Copilot) are now five-line pointers that redirect any tool back to `AGENTS.md`.
- **`docs/markdown-style.md`** — Markdown linting rules (MD001-MD047, table format, code-block languages, nested code blocks, angle-bracket placeholder rule). Pulled out so the guidance is loaded only when an agent is actually producing or editing Markdown — not on every code-only task.

### Changed

- **`ENGINEERING_PRINCIPLES.md` § 1** — language rule tightened from generic "English" to "British English (en-GB)" with concrete examples (colour / initialise / behaviour / licence vs. license).
- **`ENGINEERING_PRINCIPLES.md` § 7** — README skeleton (one-sentence pitch as blockquote, "What is this for?", use-case dialogue) absorbed into the principles. Was previously in `.github/copilot-instructions.md`; moved here because it applies to humans too.
- **`ENGINEERING_PRINCIPLES.md` (cross-refs)** — `CLAUDE.md` references updated to `AGENTS.md` throughout (§ 0, § 7, § 10, § 11). `agent:claude` label example generalised to `agent:<tool>`.
- **`CLAUDE.md`** — reduced from a project-skeleton overlay (~120 lines) to a five-line pointer at `AGENTS.md`. The skeleton's content (project facts, tech stack, overrides, licence-header examples) moved to `AGENTS.md` because that's the canonical agent brief now.
- **`.github/copilot-instructions.md`** — reduced from ~530 lines to a five-line pointer at `AGENTS.md`. The AI-specific behaviour parts moved to `AGENTS.md`; the human-relevant parts (British English, README structure) absorbed into `ENGINEERING_PRINCIPLES.md`; the Markdown lint rules moved to `docs/markdown-style.md`.
- **`README.md`** + **`docs/app-concept.md`** structure trees updated to reflect the new layout.

### Removed

- Redundant content in `.github/copilot-instructions.md` that was already in `ENGINEERING_PRINCIPLES.md` — Required-OSS-Files table, Test-Harness-First section, Testing Pyramid, File Generation Standards, Version Control Standards, Org Information header, "Excellence by Default" buzzword principles. Each of these had a canonical home elsewhere; the duplicate is gone.
- **`.github/prompts/`** (entire directory, 6 files, ~160 lines) — Copilot-Chat slash-command shortcuts (`/check-pr`, `/merge-pr`, `/create-pr`, `/new-feature`, `/auto-merge-pr`, `/add-instruction`). Each was either redundant against `AGENTS.md` / PRINCIPLES / `PULL_REQUEST_TEMPLATE.md`, an anti-pattern (admin-bypass on `gh pr merge --admin`), or referenced renamed files (`copilot-instructions.md` no longer holds content). One file even had hardcoded leftovers from another project ("Talos API Rust client library"). No XMV-OSS-relevant content was lost.
- **4 of 6 scripts in `.github/gh-scripts/`** (~400 lines): `check-pr.sh`, `create-pr.sh`, `merge-pr.sh`, `new-feature.sh`. Each was a thin wrapper around a standard `gh` command (`gh pr checks` / `gh pr create` / `gh pr merge` / `git checkout -b`) coupled to the deleted prompt files. The 2 genuinely useful bootstrap scripts (`assign-repo-to-team.sh` and `setup-branch-protection.sh`) are kept — they read `repo.ini` and configure GitHub repo settings, which is exactly the kind of operation that `ENGINEERING_PRINCIPLES.md` § 8 mandates be scripted, not run ad-hoc.

## [v0.2.0] — 2026-05-09 — initial hardening pass

A hardening pass driven by lessons learned from the first two projects bootstrapped from this template (`sharepoint-mcp`, `outlook-mcp`). Everything in this release is application- and framework-independent — the template stays language- and domain-neutral.

### Added

- **`ENGINEERING_PRINCIPLES.md`** — the 434-line project-agnostic engineering baseline (language rule, status workflow, three test layers with harness as the AI-development gate, source-control rules, CI vigilance, doc-mirrors-repo, source-of-truth, PR discipline). Both sister projects carried an identical copy; promoting it to a first-class template artefact.
- **`CLAUDE.md`** — per-project overlay skeleton: tech stack placeholder, project-specific overrides table, licence/SPDX guidance, GitHub-Projects-as-tracker pointer.
- **`docs/proposals/`** — RFC layout: a `README.md` explaining when to use it (and when *not* to), and a `_template.md` with the canonical sections (Status, Context, Decision, Alternatives considered, Consequences, Implementation notes). Naming convention `YYYY-MM-DD-short-slug.md`. Lifecycle: Draft → Accepted → Implemented; or Superseded / Withdrawn — never rewritten in place.
- **`scripts/`** — directory placeholder with a `README.md` documenting the convention from § 8 of the engineering principles (shebang + SPDX, `set -euo pipefail`, idempotent, self-documenting, `--help` summary, exit codes).
- **`.github/workflows/HARNESS_JOB.md`** — copy-paste-ready harness job snippet for downstream projects, with the canonical "skip silently when secret is missing" guard so PRs from forks aren't blocked.

### Changed

- **GitHub Issues + GitHub Projects is now the canonical tracker** for every XMV OSS project, from day one. The older "markdown TODO/ISSUES files in `docs/`" pattern is retired. `ENGINEERING_PRINCIPLES.md` § 2 rewritten; § 7 and § 10 references updated; `.github/copilot-instructions.md` and `docs/testconcept.md` updated to file issues + close via PR; `docs/app-concept.md` structure tree drops `todo.md`.
- **README skeleton** in `.github/copilot-instructions.md` now includes the three sections that paid back in both sister projects: a one-sentence pitch as a blockquote (under 200 characters, has to stand alone on package registries), a "What is this for?" section (concrete user situation before features), and a use-case dialogue example (memorable headline workflow).
- **`.github/workflows/ci.yml`** — header comment block documenting the three-job shape (lint / test / harness) and what to swap in for language-specific bits. The bats-test job for the template's own tests is unchanged.
- **`.github/workflows/release.yml`** — header comment documenting the OIDC Trusted Publisher pattern (PyPI flavour, generalises to crates.io / npm).

### Removed

- **`docs/todo.md`** — retired in favour of GitHub Issues + Projects.
